# ICMP Scanner Configuration

**File:** `settings/icmp_settings.toml`  
**Module:** ICMP Reachability Scanner (Ping Engine)

This configuration controls how the ICMP module sends echo requests (`ICMP Echo Request`) to determine whether a host is reachable.

The ICMP scanner is typically the **first stage in multi‑step pipelines**, such as:

```
ICMP → TCP SYN → HTTP → XRAY
```

Optimizing these parameters affects accuracy, speed, reliability, and network footprint.

---

## `timeout`

```toml
timeout = 2000
```

**Type:** Integer (milliseconds)  
**Default:** `2000`

Defines how long the scanner waits for an **ICMP reply** after sending an Echo Request.

### Behavior

- A target is marked **unreachable** if no reply is received within this timeout.
- Too low = false-negative results  
- Too high = slower scan

### Recommended Values

| Environment | Suggested Timeout |
|------------|-------------------|
| Local LAN / datacenter | 300–800 ms |
| Residential networks | 1000–2000 ms |
| Cross‑continent / unstable links | 2000–3000 ms |

---

## `tries`

```toml
tries = 1
```

**Type:** Integer  
**Default:** `1`

Specifies how many times the scanner retries an ICMP request if no response is received.

### Behavior

- `1` → Fastest scanning, but less reliable  
- `2–3` → Higher accuracy on unstable networks  
- `>3` → Rarely recommended (too slow)

### Reliability Impact

| Tries | Reliability | Scan Duration |
|-------|-------------|----------------|
| 1 | Low–Medium | Fast |
| 2 | Medium–High | Slightly slower |
| 3 | Very high | Significantly slower |

---

## `workers`

```toml
workers = 1000
```

**Type:** Integer  
**Default:** `1000`

Defines the number of **concurrent goroutines** sending ICMP echo requests.

Higher values increase scan throughput but also increase:

- CPU usage  
- Network bandwidth  
- Risk of triggering firewalls / rate limiters

### Recommended Worker Values

| Scan Scale | Suggested Workers |
|------------|------------------|
| Small scans (<50k IPs) | 200–500 |
| Medium scans (50k–500k IPs) | 500–1500 |
| Large-scale internet scans | 1500–3000 |
| Very aggressive / internal datacenter | 3000+ |

### Notes

- If too low → scan becomes slow  
- If too high → packet loss or ICMP throttling may occur  

---

## `shuffle_ips`

```toml
shuffle_ips = true
```

**Type:** Boolean  
**Default:** `true`

Determines whether the IP list is **randomized before scanning**.

### Why Shuffling Matters

Without shuffling:

- Sequential scans may hit entire subnets at once  
- Burst traffic appears uniform and predictable  
- Could trigger IDS/IPS and firewall protections

With shuffling enabled:

- Traffic is spread evenly  
- Load distribution improves  
- Less likely to hit rate limits

### Recommended Setting

```
true (default)
```

Should only be disabled if IP order **must** be preserved.

---

## `prefix_output`

```toml
prefix_output = "icmp_"
```

**Type:** String  
**Default:** `"icmp_"`

Sets the prefix for output files generated by the ICMP scanner.

### Use Cases

- Organizing results for multi-stage pipelines  
- Preventing conflicts when multiple scanners run simultaneously  
- Simplifying tools that parse result files

### Example Output Files

```
icmp_20260427_210734.csv
```

You can customize the prefix if multiple ICMP scans run in parallel:

```toml
prefix_output = "icmp_scan1_"
```
