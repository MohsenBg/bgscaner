# Xray Scanner Configuration

**File:** `settings/xray_settings.toml`  
**Module:** Xray Connectivity & Performance Scanner

This module performs **connectivity and performance tests using Xray** to evaluate whether a server is reachable and how well it performs under simulated traffic.

Depending on the selected configuration, the scanner can:

- Test basic connectivity
- Measure download throughput
- Measure upload throughput
- Run full bidirectional speed tests

---

# `timeout`

```toml
timeout = 6000
```

**Type:** Integer (milliseconds)  
**Default:** `6000`

Maximum time allowed for each Xray test attempt.

If the connection or test does not complete within this period, the attempt is marked as **failed**.

### Behavior

- Lower values → faster scans but higher chance of timeouts
- Higher values → more reliable but slower scanning

### Recommended Values

| Environment | Suggested Timeout |
|-------------|------------------|
| Fast datacenter networks | 3000–5000 |
| Typical internet | 6000–8000 |
| High latency environments | 8000–12000 |

---

# `workers`

```toml
workers = 32
```

**Type:** Integer  
**Default:** `32`

Number of concurrent worker goroutines running Xray tests.

Each worker performs a complete connection and optional speed test.

### Behavior

Higher values increase:

- Scan throughput
- Bandwidth usage
- CPU usage
- Xray process activity

### Recommended Values

| System Type | Suggested Workers |
|--------------|------------------|
| Low‑resource VPS | 1–16 |
| Standard server | 16–32 |
| High‑performance server | 32-64+ |

Note that speed tests can consume significant bandwidth when many workers run simultaneously.

---

# `shuffle_ips`

```toml
shuffle_ips = true
```

**Type:** Boolean  
**Default:** `true`

Randomizes the order of IP addresses before scanning.

### Benefits

- Prevents sequential scanning patterns
- Distributes traffic more evenly
- Reduces burst traffic to specific subnets
- Helps avoid triggering rate‑limiting mechanisms

Disable only when deterministic scan order is required.

---

# `connectivity_test_type`

```toml
connectivity_test_type = 3
```

**Type:** Integer  
**Default:** `3`

Defines which type of test the scanner performs.

### Available Modes

| Value | Mode | Description |
|------|------|-------------|
| `0` | ConnectivityOnly | Only checks if the server is reachable |
| `1` | DownloadSpeedOnly | Measures download throughput |
| `2` | UploadSpeedOnly | Measures upload throughput |
| `3` | Both | Performs both upload and download speed tests |

### Use Cases

**ConnectivityOnly (0)**  
Fastest mode used for filtering reachable servers.

**DownloadSpeedOnly (1)**  
Useful when evaluating streaming or content delivery performance.

**UploadSpeedOnly (2)**  
Used for testing outbound capacity or tunneling performance.

**Both (3)**  
Full performance evaluation.

---

# `download_speed`

```toml
download_speed = 100
```

**Type:** Integer (Kbps)

Target download throughput during performance tests.

Used only when:

```
connectivity_test_type = 1
or
connectivity_test_type = 3
```

### Purpose

Limits bandwidth usage while still measuring connection stability.

### Example

```
20-100 Kbps  → light traffic test
100-1000 Kbps → moderate performance test
1000+ Kbps → heavy throughput testing
```

---

# `upload_speed`

```toml
upload_speed = 50
```

**Type:** Integer (Kbps)

Target upload throughput used during upload performance testing.

Active only when:

```
connectivity_test_type = 2
or
connectivity_test_type = 3
```

### Notes

Upload tests typically consume:

- more CPU
- more outbound bandwidth

Use conservative values when scanning many servers concurrently.

# `pre_scan_type`

```toml
pre_scan_type = "tcp"
```

**Type:** String  
**Default:** `"none"`

Defines a **pre‑scan filter** executed before running the heavier Xray test.

Pre‑scans help eliminate unreachable hosts early and significantly improve overall scanning efficiency.

### Available Options

| Value | Description |
|------|-------------|
| `none` | Skip pre‑scan and run Xray directly |
| `icmp` | Perform ICMP ping test |
| `tcp` | Attempt TCP connection to target port |
| `http` | Perform HTTP/HTTPS request |

### Recommended Choice

`tcp` is usually the best option because:

- Many servers block ICMP
- TCP confirms the service port is reachable
- Faster than HTTP checks
